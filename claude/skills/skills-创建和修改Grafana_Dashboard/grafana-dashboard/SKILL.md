# 核心原则

## 1. 信息层级设计

仪表板应按重要性组织信息：
```
┌─────────────────────────────────────┐
│  关键指标 (大数字/Stat 面板)         │
├─────────────────────────────────────┤
│  关键趋势 (时间序列图)               │
├─────────────────────────────────────┤
│  详细指标 (表格/热力图)              │
└─────────────────────────────────────┘
```

## 2. 监控方法论

- **RED 方法**（服务监控）：Rate（请求率）、Errors（错误率）、Duration（延迟）
- **USE 方法**（资源监控）：Utilization（利用率）、Saturation（饱和度）、Errors（错误）

## 3. 查询优化

- 使用适当的聚合函数
- 合理设置时间窗口（如 `[5m]`）
- 利用标签过滤减少数据量

# API鉴权

## 使用用户名和密码鉴权

步骤：

1. 通过环境变量`grafana_username`获取账号，`grafana_password`获取密码。如果获取失败，尝试使用其他认证方式，或询问用户。
2. 使用HTTP Basic Authentication进行账号密码认证。

示例：

```sh
# 假设账号是admin，密码是12345
curl -u admin:12345 https://<grafana域名>/api/xxx
```

# 指定Org

- Grafana中，不同Org下的资源是隔离的，所以请求Dashboard等资源时，需要携带Org信息，否则默认是用户最后选择的Org，这个会影响判断.
- 通过Org的操作API可以获取所有Org信息。

指定Org的方式：

- 请求头Key：X-Grafana-Org-Id
- 请求头Value：org的id

## 需求分析

在创建仪表板前，确认以下信息：

1. **监控目标**：需要监控什么系统/服务？
2. **关键指标**：哪些指标最重要？
3. **目标用户**：运维人员、开发人员还是管理层？
4. **刷新频率**：实时（30s）还是定期（5m）？

## 选择仪表板类型

根据监控目标选择合适的仪表板类型：

| 类型 | 适用场景 | 关键面板 |
|------|----------|----------|
| API 监控 | HTTP 服务 | 请求速率、错误率、延迟百分位 |
| 基础设施 | 服务器/集群 | CPU、内存、磁盘、网络 |
| 数据库 | DB 服务 | QPS、连接数、慢查询 |
| 应用监控 | 业务系统 | 活跃用户、缓存命中率 |

## 设计面板布局

1. 创建 Stat 面板展示关键数字
2. 添加时间序列图展示趋势
3. 使用表格展示详细数据
4. 配置热力图分析分布

## 验证与测试

1. 在不同时间范围测试仪表板
2. 验证 PromQL 查询正确性
3. 确认告警触发条件

# 最佳实践

## 设计原则

1. **从模板开始**：使用 Grafana 社区仪表板作为起点
2. **一致命名**：面板和变量使用统一命名规范
3. **分组展示**：将相关指标组织在同一个 Row 中
4. **合理默认**：默认时间范围设为最近 6 小时
5. **变量驱动**：使用变量提高仪表板灵活性
6. **添加说明**：为复杂面板添加描述信息
7. **单位正确**：为指标配置合适的显示单位
8. **阈值明确**：设置合理的颜色阈值
9. **颜色一致**：跨仪表板保持颜色语义一致
10. **充分测试**：在不同时间范围验证显示效果

## 面板类型选择

| 场景 | 推荐面板 | 示例 |
|------|----------|------|
| 单值指标 | Stat | 总请求数、错误率 |
| 趋势分析 | Time Series | CPU 使用率、请求量 |
| 详细数据 | Table | 服务状态列表 |
| 分布分析 | Heatmap | 延迟分布 |

# 注意事项

* **PromQL 语法**：确保查询语法正确，使用反引号包裹字段名
* **时间窗口**：根据指标类型选择合适的窗口大小（如 `[5m]`、`[1h]`）
* **标签基数**：避免使用高基数标签导致查询性能问题
* **权限配置**：生产仪表板需要配置适当的访问权限
* **版本管理**：使用 Dashboard as Code 进行版本控制
* **资源引用**：确保参考的 assets 文件存在
